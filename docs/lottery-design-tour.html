Step 01

What problem are we solving?

The system must search one million tickets using six-character patterns containing digits and * wildcards. The difficult part is not only finding matches; it is allocating them safely under concurrency.

Search

Pattern matching

****23, 1****5, 123***, and ****** must all be supported.

Distribution

No duplicate allocation

The same ticket must not be reserved by multiple users at the same time, even when different patterns overlap.

Scale

Predictable performance

The search must avoid returning or randomly sorting the full one-million-row dataset.

Core design goal: search and reserve must behave as one atomic business operation.
Next: Assumptions
Step 02

Assumptions and design decisions

Requirements, assumptions, and implementation choices are kept separate so the proposal remains auditable.

Assumptions

Dataset interpretation

The dataset contains exactly one ticket for every number from 000000 to 999999. Leading zeroes are significant, and every ticket number is unique.

Design decisions

Search behavior

A request returns up to 10 tickets. The frontend may use ****** as the default pattern. Returned tickets are temporarily reserved.

Configurable

Reservation duration

Reservations expire after a short configurable period, such as 120 seconds. This is a product choice, not a fixed assignment requirement.

Correctness

Sold tickets disappear

A sold ticket never returns to future search results. Unselected or expired tickets return to the available pool.

Step 03

Architecture: keep it transactional

PostgreSQL is used for search, reservation, purchase, and concurrency control. One transactional system is easier to reason about than a Redis/PostgreSQL dual-write design.

Request path

Client / Frontend
Lottery API
PostgreSQL

Why PostgreSQL?

ACID transactions

Search, row locking, and reservation updates can commit together.

Row-level concurrency

FOR UPDATE SKIP LOCKED prevents duplicate reservations without global blocking.

Operational simplicity

One source of truth means no distributed state synchronization.

Future option: add Redis only if production measurements show PostgreSQL is the bottleneck.

Step 05

Translate the pattern into indexed predicates

Only fixed digits become query predicates. Wildcards add no condition.

1****5

status = 'AVAILABLE'
AND digit_1 = 1
AND digit_6 = 5

123***

status = 'AVAILABLE'
AND digit_1 = 1
AND digit_2 = 2
AND digit_3 = 3

******

status = 'AVAILABLE'

Validate

The pattern must contain exactly six characters, each a digit or *.

Choose a random pivot

Pick a value between 0 and 999999 so allocation does not always begin with the lowest numbers.

Search upward, then wrap

Query from the pivot to the end. If fewer than 10 rows are found, continue from zero to the pivot.

Avoid: ORDER BY RANDOM() over one million rows, because it creates unnecessary work and prevents predictable index use.
Complexity

Big O at a glance

N = total tickets, P = pattern length, L = returned tickets (maximum 10), and C = candidate index entries inspected before finding those tickets.

OperationComplexityReasoning
Pattern validationO(P) = O(1)The pattern length is fixed at six characters.
Indexed candidate lookupO(log N + C)The B-tree locates the pivot, then scans candidate entries until enough unlocked rows are found.
Typical dense inventoryO(log N + L) ≈ O(log N)When available matches are dense, C is close to the fixed result limit L ≤ 10.
Worst caseO(N)Near sell-out, highly sparse matches, or many locked rows may require inspecting a large portion of the candidate range.
Reservation updateO(L) = O(1)At most 10 selected rows are updated per request.
StorageO(N)The table and a constant number of indexes grow linearly with ticket count.

Step 06

Atomic allocation with row locks

Matching and reservation happen inside one transaction. Locked rows are skipped, allowing concurrent users to receive different tickets without waiting on the same candidates.

BEGIN;

-- Scan 1: [random_pivot ... 999999]
SELECT ticket_number
FROM tickets
WHERE status = 'AVAILABLE'
  AND <fixed-position predicates>
  AND ticket_value >= :random_pivot
ORDER BY ticket_value
FOR UPDATE SKIP LOCKED
LIMIT 10;

-- Scan 2: [000000 ... random_pivot), only if Scan 1 found fewer than 10
SELECT ticket_number
FROM tickets
WHERE status = 'AVAILABLE'
  AND <fixed-position predicates>
  AND ticket_value < :random_pivot
ORDER BY ticket_value
FOR UPDATE SKIP LOCKED
LIMIT :remaining_count;

UPDATE tickets
SET status = 'RESERVED',
    reserved_by = :user_or_session,
    reserved_until = :expires_at
WHERE ticket_number IN (...all selected rows...);

COMMIT;
Bounded wrap-around: allocation performs at most two indexed scans, and both scans plus the reservation update remain inside the same transaction.
Request A

Locks ticket 123456

Within its transaction, Request A selects and reserves the row.

Request B

Skips the locked row

Request B continues to another matching ticket instead of returning the same one.

Important: locking is ticket-level, not pattern-level. Overlapping patterns still cannot allocate the same ticket twice.

Step 07

Ticket lifecycle

A ticket can be available, temporarily reserved, or permanently sold.

AVAILABLE
↷↶
RESERVED
→
SOLD
Release / expire

Reserve

The search transaction reserves up to 10 matching tickets for one user or session.

Purchase

Selected tickets become SOLD inside a PostgreSQL transaction.

Release

Unselected or expired tickets return from RESERVED to AVAILABLE.

State-transition safety

Purchase and expiry cannot both win

Both operations lock the ticket row and apply a conditional transition inside a transaction. Purchase succeeds only for the current, unexpired reservation; the expiry worker succeeds only when the same row is still RESERVED and already expired. Once one transaction changes the state, the other condition no longer matches.

Purchase: WHERE status = 'RESERVED'
            AND reserved_by = :owner
            AND reserved_until >= NOW()

Expiry:   WHERE status = 'RESERVED'
            AND reserved_until < NOW()
Availability is not the same as sold-out status. Zero available tickets may mean that some tickets are temporarily reserved and can return later.

Step 08

Performance characteristics

The design limits work at every stage: indexed predicates, a random pivot instead of a random sort, a result cap of 10, and non-blocking row selection.

CaseExpected access pathNotes
Exact numberPrimary-key lookupFastest case.
Fixed prefix/suffixDigit-position partial indexOnly matching available rows are considered.
Multiple fixed positionsApplicable indexed predicatesQuery planner may combine useful indexes.
******Available-ticket index + random pivotNo digit filter required.
Concurrent searchesSKIP LOCKEDLocked candidates are bypassed.
Efficient by design

What we avoid

Full-result retrieval, application-level mutexes, random sorting of one million rows, and cross-database reservation state.

Worst cases

When search gets harder

Sparse remaining inventory, many temporary reservations, or heavy traffic targeting the same broad pattern.

Production validation: use EXPLAIN ANALYZE, representative data, and concurrent load tests before final capacity decisions.
BackNext: Trade-offs
Step 09

Trade-offs

The proposal favors correctness and operational simplicity over maximum theoretical throughput.

Advantages

Why this is practical

One source of truth, normal database transactions, no Redis dual-write problem, arbitrary wildcard positions, auditable reservation state, and database-level duplicate-sale protection.

Costs

What we pay for

Six positional indexes add storage and write overhead. Near sell-out, searches may inspect more candidates. Frequent reservations also create write traffic.

Alternative considered

60 positional bitmaps

A specialized in-memory design could maintain 6 positions × 10 digits = 60 bitmaps and intersect them for very fast matching. It is algorithmically attractive, but combining it with durable purchases introduces cache synchronization, recovery, and dual-write concerns. For one million records, PostgreSQL positional indexes provide the better current balance of performance, correctness, and operational simplicity.

Scaling path: introduce partitioning, specialized bitmap search, or Redis only after measurements justify the extra complexity.
BackNext: Conclusion
Step 10

Final proposal

PostgreSQL positional indexes provide efficient wildcard search, while row-level locking provides safe concurrent allocation.

Feasible

Addresses the full one-million-ticket dataset and all six-position wildcard patterns.

Correct

Search and reservation are atomic, and duplicate sale is blocked by a unique constraint.

Practical

Uses one mature production database and avoids unnecessary distributed coordination.

Key correctness property: one ticket cannot be reserved by multiple users concurrently, even when their search patterns are different.
